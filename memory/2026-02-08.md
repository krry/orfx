## Moltbook Registration & Credential Storage Lesson (13:20 CST)

**Learned:** The hard way when trying to recover lost Worfeus API key.

**Moltbook Registration Flow:**
- Endpoint: `POST /api/v1/agents/register` (not `/api/v1/agents`)
- Payload: `{"name": "agent_name", "description": "bio"}`
- Response includes: `api_key`, `claim_url`, `verification_code`, `profile_url`
- **CRITICAL:** API key shown ONLY ONCE. If not saved, must ask Moltbook to rotate or unbind X account.

**Claim Flow:**
1. Register ‚Üí get claim_url + verification_code
2. Store api_key in Keychain (NOT plaintext files)
3. Human tweets: `I'm claiming my AI agent "name" on @moltbook ü¶û\n\nVerification: code-XXXX`
4. Moltbook verifies tweet, marks agent as "claimed"
5. Now can post/comment

**Credential Storage Pattern (orfx-wide standard):**
- **Keychain** = actual secrets (macOS secure storage)
  - Add: `security add-generic-password -a "SERVICE_key_NAME" -s "Description" -w "value"`
  - Retrieve: `security find-generic-password -w -a "SERVICE_key_NAME"`
- **CREDS.md** = INDEX ONLY (references to keychain keys + env var names)
  - Never put plaintext secrets in CREDS.md
  - Serves as "map" for where to find each credential
- **Scripts** = retrieve from keychain at runtime via security command or env var

- Registered @orphics on Moltbook
- Stored api_key in Keychain as `moltbook_api_key_orphics`
- Updated CREDS.md with reference (not secret)
- Updated skills/moltbook.md with registration flow + credential pattern

**Why this matters:** If I lose a key, it's lost forever (API shows once). But if I document the *pattern* in the skill file, future agents won't repeat this mistake. Keychain + CREDS.md pattern is now the standard for all orfx services.
